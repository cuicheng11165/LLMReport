> 说明：这是实战选型导向的高层对比，不是严格基准榜单。建议最终以你的任务集做 A/B 测试。

## 1) 快速结论

- **代码优先**：Claude（Sonnet/Opus）、GPT（4.x/o 系列）、Codestral、Qwen-Coder
- **中文优先**：Qwen、GLM、Kimi、混元、文心（按场景差异较大）
- **成本优先（API）**：开源自建（Llama/Qwen/Mistral/Yi）通常长期更低；托管 API 则看 token 单价与缓存策略
- **综合均衡**：先用 2~3 家并行评测，再收敛到 1 主 1 备

## 3) 按目标选型建议

### A. 你要“代码生产力最大化”
- 首选候选：Claude + GPT + Qwen-Coder
- 评测任务：
  1. 读现有仓库并定位 bug
  2. 新增一个 API + 测试
  3. 代码审查与重构
- 关键指标：一次通过率、修复回归率、代码可读性、总 token 成本

### B. 你要“中文业务对话/文档能力”
- 首选候选：Qwen + GLM + Kimi（加 1 个国际模型做对照）
- 评测任务：
  1. 中文客服问答
  2. 长文总结与信息抽取
  3. 专业术语一致性
- 关键指标：事实准确率、术语一致性、幻觉率、延迟

### C. 你要“成本最优”
- 思路：
  - 高频请求走轻量模型
  - 关键请求升级高阶模型
  - 启用缓存、截断、RAG 减少无效 token
- 常见架构：
  - Router（轻量分类）→ 小模型（默认）→ 大模型（困难样本）

## 4) 成本控制清单（可直接落地）

1. 建立 **prompt 模板**，减少冗余上下文
2. 开启 **结果缓存**（尤其重复问答）
3. RAG 只注入必要片段，避免“整库塞入”
4. 按任务分级路由模型（cheap-first）
5. 每周看一次：
   - 成本/成功率
   - 平均延迟
   - 用户满意度
